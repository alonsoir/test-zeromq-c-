    ADR-024 — Dynamic Group Key Agreement (Noise_IKpsk3)

    Status: DISEÑO APROBADO — OQs CERRADAS — IMPLEMENTACIÓN DESBLOQUEADA Date: 2026-04-01 (DAY 104) OQs cerradas: 2026-04-12 (DAY 115) — Consejo de Sabios Branch implementación: feature/adr024-noise-p2p Consejo de Sabios Ronda 1: DAY 103 — Claude, Grok, ChatGPT, DeepSeek, Gemini Consejo de Sabios Ronda 2: DAY 104 — ChatGPT, DeepSeek, Gemini, Grok, Qwen — unanimidad Consejo de Sabios OQ-5..8: DAY 115 — ChatGPT, DeepSeek, Gemini, Grok, Qwen, Claude — unanimidad Deciders: Alonso Isidoro Roman (árbitro — se adhiere al consenso del Consejo DAY 115)

    Context

    The current ML Defender cryptographic transport (ADR-013, ADR-015, ADR-022) derives per-channel subkeys via HKDF-SHA256 from a 32-byte seed (seed.bin) provisioned statically at deployment time by tools/provision.sh. All six pipeline components on a single host share a common seed family.

    This design has two acknowledged limitations (documented in §10.11 of the paper):

    1. Seed compromise — if seed.bin is exfiltrated, an adversary can derive all component subkeys and forge authenticated messages.
    2. Single-host key assumption — no mechanism for two independently provisioned ML Defender nodes to establish a shared session key without pre-sharing a seed out-of-band.

    ADR-024 designs a solution using the Noise Protocol Framework (Noise_IKpsk3 handshake pattern) combined with libsodium 1.0.19 and the existing seed infrastructure as a Pre-Shared Key (PSK) source.

    Decision

    Adopt Noise_IKpsk3 as the dynamic key agreement handshake for ML Defender's inter-component and inter-node channel establishment.

    Key Derivation Chain and Domain Separation (D4)

    seed_family (32 bytes, from ADR-021 deployment.yml)
    │
    └─ HKDF-SHA256(seed_family, info="ml-defender:noise-ikpsk3:v1")
         │
         └─ PSK (32 bytes)  ← injected into Noise_IKpsk3 handshake as psk3

    Domain separation is mandatory. The canonical approved info string is:

    "ml-defender:noise-ikpsk3:v1"

    Prohibited info strings (R8)

    Prohibited string Reason
    "noise-ik-psk" Too generic — collision risk with other uses
    "ml-defender:noise" Missing version — prevents key rotation
    "" (empty) HKDF anti-pattern — weakens domain separation

    Security implication of seed_family compromise (R7)

    Compromise of seed_family allows derivation of the Noise PSK, enabling an attacker to impersonate any component in future handshakes. Forward secrecy of past sessions is preserved (ephemeral keys are not derived from seed_family). Rotation of seed_family requires reprovisioning all components (tools/provision.sh). See OQ-5 resolution below.

    Handshake Pattern: Noise_IKpsk3

    Pattern: Noise_IKpsk3

Pre-message:
  → s   (responder's static public key, known to initiator at deploy time)

Handshake (1-RTT, on component startup only):
  → e, es, s, ss
  ← e, ee, se, psk

    Properties: - 1-RTT — completes in one round trip; suitable for startup sequences. - Identity hiding — initiator's static key s is encrypted under the responder's static key in the first message. - PSK binding — psk3 mixes the HKDF-derived PSK into the final handshake hash, binding the session to the deployment's seed family. - Forward secrecy — ephemeral e/ee exchange provides per-session forward secrecy.

    Implementation Stack

    Layer Component Version
    Noise handshake noise-c commit-pinned (vendored)
    Crypto primitives libsodium 1.0.19 (compiled from source)
    PSK derivation HKDF-SHA256 libsodium crypto_kdf_hkdf_sha256
    Static keypairs X25519 generated by extended tools/provision.sh
    Transport after handshake ChaCha20-Poly1305 existing CryptoTransport

    Fallback (R10): If noise-c binary footprint exceeds 200 KB on target ARMv8, implement Noise_IKpsk3 directly over libsodium (X25519 + HKDF-SHA256 + ChaCha20-Poly1305). Both paths are cryptographically equivalent; libsodium-direct loses formal Noise framework validation but no security properties.

    noise-c acceptance criteria (R10) — mandatory benchmark before production

    • Binary footprint: < 200 KB
    • Handshake latency on ARMv8 Cortex-A72 (Raspberry Pi 4): < 50 ms

    Analytical estimate (Consejo DAY 115 consensus): X25519 ≈ 1–3 ms, ChaCha20-Poly1305 ≈ 0.1–0.2 ms, HKDF negligible → total handshake << 10 ms. The 50 ms threshold is conservative; bottleneck will be network, not crypto.

    Integration with CryptoTransport: Atomic Key Installation (D6)

    // Called once per component startup, after Noise handshake completes.
// Thread-safe: internal mutex prevents concurrent access.
void CryptoTransport::install_session_keys(
    const uint8_t tx_key[32],
    const uint8_t rx_key[32]
);

    Contract: - Called exactly once per session, post-handshake. - Atomic: uses an internal mutex. No observable intermediate state. - Component does not process pipeline messages before install_session_keys() completes. Gate is etcd READY registration. - If called more than once: std::terminate().

    Handshake Timing and Failure Semantics

    1. Component starts → reads seed.bin via seed-client.
    2. HKDF derives PSK from seed_family using approved info string.
    3. Component initiates (or accepts) Noise_IKpsk3 handshake with its peer.
    4. install_session_keys() installs session keys into CryptoTransport.
    5. Component registers with etcd as READY.
    6. Normal pipeline operation begins.

    Failure policy: retry up to 5 times with 2-second intervals. All retries fail or PSK/key mismatch detected: std::terminate() (fail-closed).

    Static Keypair Provisioning

    tools/provision.sh extended to generate a dedicated X25519 static keypair per component. Separate from Ed25519 keypairs used for log signing (ADR-013). Key reuse across different cryptographic contexts is explicitly prohibited.

    Static public keys stored in deployment.yml (ADR-021) as the Noise pre-message payload. deployment.yml is extended with allowed_static_keys per peer (see OQ-5 resolution).

    Resolved Open Questions

    OQ-5 — Static Key Revocation ✅ CERRADA DAY 115

    Veredicto Consejo (6/6 consenso): IMPLEMENTAR allowed_static_keys en deployment.yml con hot-reload vía etcd.

    Decisión: - deployment.yml incluye una lista allowed_static_keys por peer (fingerprints X25519 autorizados). - Cada componente mantiene la lista en caché local en memoria — sin query a etcd en la ruta crítica de handshake. - La lista se actualiza vía hot-reload al detectar nueva versión en etcd. - etcd es canal de propagación, no root of trust. La lista debe estar firmada con clave derivada de seed_family. - Si seed_family es exfiltrada: re-provision completo obligatorio (no hay mecanismo de revocación que sobreviva a la pérdida del PSK root).

    Procedimiento de emergencia (robo de nodo): 1. Detectar compromiso. 2. Eliminar fingerprint del nodo robado de allowed_static_keys en deployment.yml. 3. Distribuir via etcd → todos los componentes rechazan handshakes del nodo robado. 4. En ventana de mantenimiento: rotar seed_family + regenerar todos los keypairs X25519 (provision.sh completo).

    Riesgo residual aceptado: Ventana entre robo físico y ejecución del procedimiento de emergencia.

    Posición minoritaria registrada: Blocklist-only vía etcd sin caché local (ChatGPT) — rechazada por dependencia circular si etcd comprometido.

    OQ-6 — Key Rotation Zero Downtime ✅ CERRADA DAY 115

    Veredicto Consejo (6/6 consenso): IMPLEMENTAR dual-key window T=24h + versioned deployment.yml.

    Decisión: - Ventana de gracia T = 24 horas: cada componente acepta keypair antiguo Y nuevo durante la transición. - deployment.yml versionado: la nueva clave pública se añade antes de activarla. - No se requiere coordinador (opción C rechazada: SPOF + complejidad innecesaria).

    Secuencia de rotación (cero downtime, 6 componentes):

    Paso 1: Generar nuevo keypair X25519 offline para el componente a rotar.
Paso 2: Publicar deployment.yml versión N+1 con dual pubkey (old + new) para ese peer.
Paso 3: etcd distribuye → todos los componentes recargan config en próximo heartbeat.
Paso 4: Componente rotado activa nueva private key inmediatamente.
Paso 5: Tras T=24h, publicar deployment.yml versión N+2 con single new key.
        La clave antigua queda invalidada.

    Restricción anti-downgrade: Un componente nunca acepta una clave más antigua que la actualmente registrada como current. La ventana es única por transición.

    Riesgo residual aceptado: Durante T=24h, la clave antigua sigue siendo válida. Impacto práctico bajo en LAN hospitalaria cerrada sin adversario externo.

    OQ-7 — Replay Protection First Handshake Message ✅ CERRADA DAY 115

    Veredicto Consejo (5/6 ACEPTAR, 1/6 IMPLEMENTAR): ACEPTAR riesgo documentado v1.

    Decisión: - El primer mensaje (→ e, es, s, ss) es técnicamente replayable pero el PSK binding es protección suficiente: sin seed_family correcto el handshake no puede completarse. - El riesgo real es DoS-CPU menor (handshakes fallidos que consumen CPU), no compromiso de confidencialidad ni autenticación. - Mitigación v1: rate-limiting a nivel de red via nftables/iptables (límite de handshakes por IP/minuto). Capa de red, no protocolo. - No se introduce timestamp en payload v1: requeriría NTP fiable (no garantizado en LAN hospitalaria aislada) y añade complejidad sin beneficio proporcional.

    Documentación obligatoria en threat model (§10.11 paper):

    "The first Noise_IKpsk3 handshake message is technically replayable. Without knowledge of seed_family (PSK root), replay cannot complete the handshake or compromise session keys. Risk: minor CPU DoS. Mitigation: nftables rate-limiting. This risk is accepted for v1 in closed LAN deployments."

    Posición minoritaria registrada (ChatGPT): IMPLEMENTAR timestamp + nonce aleatorio en payload con ventana ±2-5 min. Registrada para v2 si el modelo de amenaza evoluciona a WAN/VPN.

    Trigger para v2: Si aRGus NDR añade conectividad WAN o VPN, la mitigación timestamp pasa de opcional a obligatoria.

    OQ-8 — ARMv8 Performance + Noise_IKpsk3 vs Noise_KK ✅ CERRADA DAY 115

    Veredicto Consejo (6/6 consenso): MANTENER Noise_IKpsk3 para v1.

    Decisión: - Noise_IKpsk3 se mantiene como patrón de v1. PSK binding nativo (psk3) alinea directamente con la arquitectura seed_family del proyecto. - Identity hiding del iniciador: beneficio marginal en LAN cerrada, pero sin coste adicional de rendimiento. No justifica cambiar a Noise_KK. - KCI resistance de Noise_KK: no es argumento suficiente en el modelo de amenaza actual. Si un keypair estático se compromete, el vector de ataque ya está cubierto por el procedimiento de revocación (OQ-5).

    Benchmark ARMv8 obligatorio antes de producción (R10):

    Criterio Límite Estimación analítica Consejo
    noise-c binary footprint < 200 KB Por medir
    Handshake latency (Cortex-A72) < 50 ms 3–10 ms (holgado)

    Si noise-c > 200 KB: implementación directa sobre libsodium (X25519 + HKDF-SHA256 + ChaCha20-Poly1305). Criptográficamente equivalente. Se pierde: validación formal del framework Noise + auditoría estándar. Se recomienda revisión de código exhaustiva si se toma esta ruta.

    Posición minoritaria registrada (Grok): Noise_KK más simple y mejor KCI para despliegues cerrados. Reevaluar si benchmarks muestran ventaja significativa de KK. Registrado para evaluación post-benchmark.

    Riesgo Decisión Mitigación

    Nodo físico robado Mitigado allowed_static_keys + procedimiento emergencia

    Recovery Contract

    ## Recovery Contract (ADR-024)

### Compromiso de clave estática X25519 (nodo robado):
1. Eliminar fingerprint de allowed_static_keys en deployment.yml → etcd distribuye.
2. Todos los componentes rechazan handshakes del nodo robado inmediatamente.
3. En ventana de mantenimiento: provision.sh completo en nodos sanos.
4. Tiempo estimado respuesta inmediata: < 5 minutos.

### Compromiso de seed_family:
1. Asumir compromiso total del pipeline.
2. provision.sh completo en TODOS los nodos: nuevo seed_family + nuevos keypairs.
3. Tiempo estimado recuperación completa: < 30 minutos con provision.sh automatizado.
4. No hay atajo: la pérdida de seed_family invalida todo el root of trust.

    • Plugin API (plugin_process_message, ADR-023) — unchanged.

    Relationship to §10.11 Threats to Validity

    Threat (paper §10.11) Mitigation via ADR-024
    Seed compromise PSK binding + forward secrecy: past sessions protected
    Single-host key assumption Noise_IKpsk3 enables independently provisioned nodes
    Static key compromise (stolen node) allowed_static_keys + emergency re-provision
    No hardware root of trust Not addressed — TPM/secure enclave remains future work
