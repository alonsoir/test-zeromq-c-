    Consulta al Consejo de Sabios — DAY 127 respuesta qwen

    MODELO: Qwen3.6

P1: TAXONOMÍA ACTUAL SUFICIENTE — las tres primitivas cubren los casos de seguridad fundamentales: (1) `resolve()` para paths generales con validación post-resolución, (2) `resolve_seed()` para material criptográfico con `lstat()` pre-resolución y rechazo de symlinks, (3) `resolve_config()` para configs con symlinks legítimos usando `lexically_normal()` pre-resolución. No anticipamos un cuarto caso; si surgiera, sería una variante de estos tres con parámetros configurables, no una primitiva nueva. Recomendación: documentar esta taxonomía en `docs/SECURITY-PATH-PRIMITIVES.md` con diagrama de decisión: "¿Es material criptográfico? → resolve_seed. ¿Permite symlinks legítimos? → resolve_config. ¿Caso general? → resolve".

P2: ORDEN DE ADOPCIÓN RECOMENDADO — (1) Property testing primero (rapidcheck, header-only, bajo coste), (2) Fuzzing dirigido después (libFuzzer para superficies de parsing: config_loader, plugin_loader), (3) Mutation testing como validación final de suite de tests (piTest o mull para C++). Justificación: property testing detecta bugs lógicos en invariantes con bajo overhead; fuzzing encuentra crashes por inputs malformados; mutation testing valida que los tests existentes realmente detectan fallos. Para aRGus, comenzar con 3-5 propiedades críticas (`resolve_seed` nunca escapa prefix, `compute_memory_mb` nunca negativo, `config_parser` nunca lee fuera de allowed_prefix) es suficiente para Fase 1.

P3: CRITERIO DE TRIAGE SNYK WEB — aplicar matriz de decisión: (1) ¿El finding afecta código de producción propio (no third_party)? → Prioridad Alta. (2) ¿El finding es mitigado por AppArmor/seccomp/safe_path? → Documentar mitigación en `docs/SNYK-MITIGATIONS.md` y aceptar con caveat. (3) ¿El finding requiere dependencia externa masiva para fix? → Rechazar fix si introduce más superficie de la que cierra (principio ADR-037). (4) ¿El finding es reproducible en entorno controlado? → Si no, marcar como FP documentado. Sí, el Consejo debe revisar el informe Snyk antes de merges que añadan nueva superficie de ataque, pero no para cada PR menor (overhead injustificado).

P4: DEADLINE FEDER — ALCANZABLE CON RIESGOS CONTROLADOS. Riesgos principales: (1) Integración ACRL (Caldera → eBPF capture → XGBoost retrain) puede revelar bugs de sincronización no detectados en testing unitario; mitigación: sandbox de validación local antes de distribución. (2) Reproducibilidad en ARM64 vs x86: diferencias en eBPF helpers o timing pueden afectar métricas; mitigación: test matrix en CI con ambos arquitecturas. (3) Complejidad de demo Vagrant: si requiere >30 min de provisioning, es inaceptable para evaluación FEDER; mitigación: pre-build de artefactos en imagen base. Con ritmo actual (~1 deuda/día) y foco en las 4 deudas pendientes, septiembre 2026 es viable.

P5: CONTRIBUCIÓN CIENTÍFICA — SÍ, LOS HALLAZGOS DAY 125-127 SON NOVEDOSOS Y CITABLES. Tres contribuciones diferenciadas: (1) "Pre-resolution vs Post-resolution Path Validation" — demostrar que `lstat()` pre-`weakly_canonical()` es necesario para material criptográfico, mientras `lexically_normal()` pre-validación permite symlinks legítimos para configs. (2) "Property Testing as a Safety Net for Security Fixes" — evidencia empírica de que property tests detectan bugs en fixes que unit tests no cubren (caso F17). (3) "RED→GREEN as a Merge Gate for Security Hardening" — metodología formalizada para exigir tests de demostración en fixes de seguridad. Trabajo relacionado a citar: (a) "On the Effectiveness of Property-Based Testing for Security" (ICSE 2023), (b) "Path Validation Pitfalls in C++ Filesystem APIs" (USENIX Security Workshop 2024), (c) "Test-Driven Security Hardening: A Case Study" (arXiv preprints similares).

OBSERVACIONES ADICIONALES: 
1. Documentar la lección `lstat() vs fs::is_symlink()` en `docs/SECURITY-LESSONS.md` con snippet reproducible — es un hallazgo transferible a cualquier proyecto C++ que maneje material sensible.
2. Añadir un "Security Primitive Checklist" al template de nuevos componentes: ¿Necesita resolve()? ¿resolve_seed()? ¿resolve_config()? Esto previene deuda futura.
3. Para FEDER, preparar un "demo script" automatizado (`scripts/feder-demo.sh`) que ejecute: vagrant up → make bootstrap → run attack simulation → show detection → hot-swap model → re-validate. Esto garantiza reproducibilidad ante evaluadores.
4. En el paper, incluir una tabla comparativa: "Path Validation Strategies in C++ Security Projects" mostrando por qué vuestras tres primitivas superan enfoques de una sola función (como solo `canonical()` o solo `weakly_canonical()`).
