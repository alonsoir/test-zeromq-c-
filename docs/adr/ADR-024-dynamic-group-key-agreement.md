## Context

The current ML Defender cryptographic transport (ADR-013, ADR-015, ADR-022) derives
per-channel subkeys via HKDF-SHA256 from a 32-byte seed (`seed.bin`) provisioned
statically at deployment time by `tools/provision.sh`. All six pipeline components
on a single host share a common seed family.

This design has two acknowledged limitations (documented in §10.11 of the paper):

1. **Seed compromise** — if `seed.bin` is exfiltrated, an adversary can derive all
   component subkeys and forge authenticated messages.
2. **Single-host key assumption** — no mechanism for two independently provisioned
   ML Defender nodes to establish a shared session key without pre-sharing a seed
   out-of-band.

ADR-024 designs a solution using the **Noise Protocol Framework** (**Noise_IKpsk3**
handshake pattern) combined with libsodium 1.0.19 and the existing seed infrastructure
as a Pre-Shared Key (PSK) source.

**Implementation is explicitly deferred to post-arXiv.** This ADR records the design
so that implementation can proceed without a new design review cycle.

---

## Decision

Adopt **Noise_IKpsk3** as the dynamic key agreement handshake for ML Defender's
inter-component and inter-node channel establishment.

---

## Key Derivation Chain and Domain Separation (D4)

```
seed_family (32 bytes, from ADR-021 deployment.yml)
    │
    └─ HKDF-SHA256(seed_family, info="ml-defender:noise-ikpsk3:v1")
         │
         └─ PSK (32 bytes)  ← injected into Noise_IKpsk3 handshake as psk3
```

**Domain separation is mandatory.** The canonical approved info string is:

```
"ml-defender:noise-ikpsk3:v1"
```

### Prohibited info strings (R8)

The following must never be used for Noise PSK derivation:

| Prohibited string | Reason |
|-------------------|--------|
| `"noise-ik-psk"` | Too generic — collision risk with other uses |
| `"ml-defender:noise"` | Missing version — prevents key rotation |
| `""` (empty) | HKDF anti-pattern — weakens domain separation |

This derivation is the **exclusive use of `seed_family` for Noise PSK.**
Channel subkey derivation uses distinct info strings per `contexts.hpp`.
No other HKDF derivation from `seed_family` is permitted without a new ADR.

### Security implication of seed_family compromise (R7)

Compromise of `seed_family` allows derivation of the Noise PSK, enabling an attacker
to impersonate any component in **future** handshakes. Forward secrecy of **past**
sessions is preserved (ephemeral keys are not derived from `seed_family`). Rotation
of `seed_family` requires reprovisioning all components (`tools/provision.sh`) and is
addressed in OQ-6 below.

## Handshake Pattern: Noise_IKpsk3

```
Pattern: Noise_IKpsk3

Pre-message:
  → s   (responder's static public key, known to initiator at deploy time)

Handshake (1-RTT, on component startup only):
  → e, es, s, ss
  ← e, ee, se, psk
```

**Properties:**
- **1-RTT** — completes in one round trip; suitable for startup sequences.
- **Identity hiding** — initiator's static key `s` is encrypted under the responder's
  static key in the first message.
- **PSK binding** — `psk3` mixes the HKDF-derived PSK into the final handshake hash,
  binding the session to the deployment's seed family.
- **Forward secrecy** — ephemeral `e`/`ee` exchange provides per-session forward secrecy.

**Note on replay (OQ-7 — updated):**
The first handshake message (`→ e, es, s, ss`) does not carry an explicit freshness
nonce. It is technically replayable, but without knowledge of the PSK (derived from
`seed_family`) a replay is useless to an attacker — the handshake cannot complete.
The threat model therefore assumes `seed_family` is not compromised; if it is, the
entire system is compromised (see R7). An optional mitigation — including a timestamp
or challenge nonce in the first message payload (Noise permits payload in handshake
messages) — is deferred to PHASE 3 evaluation.

## Implementation Stack

| Layer | Component | Version |
|-------|-----------|---------|
| Noise handshake | noise-c | commit-pinned (vendored) |
| Crypto primitives | libsodium | 1.0.19 (compiled from source) |
| PSK derivation | HKDF-SHA256 | libsodium `crypto_kdf_hkdf_sha256` |
| Static keypairs | X25519 | generated by extended `tools/provision.sh` |
| Transport after handshake | ChaCha20-Poly1305 | existing CryptoTransport |

**Rationale for noise-c:** plain C, MIT license, no runtime dependency, compatible
with C++20, commit-pinnable for reproducible builds. Compile with hardening flags:
`-fstack-protector-strong`; ASAN in Debug builds.

**Rationale for libsodium 1.0.19:** already compiled from source (ADR-013). No new
dependency introduced for PSK derivation.

### noise-c evaluation criteria post-arXiv (R10)

Before committing to noise-c in production, measure on target hardware:
- Binary footprint increase: **< 200 KB** (otherwise: implement Noise_IKpsk3 directly
  over libsodium primitives — X25519 + HKDF-SHA256 + ChaCha20-Poly1305).
- Handshake latency: **< 50 ms** on commodity ARMv8 hardware (~150–200 USD).

If either criterion is not met, implement Noise_IKpsk3 directly over libsodium
without noise-c (estimated effort: ~2 days). Both paths are cryptographically
equivalent.

## Integration with CryptoTransport: Atomic Key Installation (D6)

Noise_IKpsk3 produces a pair of 32-byte session keys at handshake completion.
These replace the static HKDF-derived subkeys for the duration of the session.

### `install_session_keys()` specification

```cpp
// Called once per component startup, after Noise handshake completes.
// Thread-safe: internal mutex prevents concurrent access.
void CryptoTransport::install_session_keys(
    const uint8_t tx_key[32],
    const uint8_t rx_key[32]
);
```

**Contract:**
- Called **exactly once** per session, post-handshake.
- Atomic: uses an internal mutex. No observable intermediate state.
- The component does **not** process pipeline messages before `install_session_keys()`
  completes. The gate is etcd `READY` registration, which is not emitted until
  after successful key installation.
- If `install_session_keys()` is called more than once, the second call invokes
  `std::terminate()`.

**`MessageContext` (ADR-023) is unchanged from the plugin's perspective.** The
`channel_id` and `direction` fields continue to identify the logical channel; the
underlying keys are now session-derived rather than static.

## Handshake Timing and Failure Semantics

The handshake occurs **once per component startup**, at the etcd registration point:

1. Component starts → reads `seed.bin` via `seed-client`.
2. HKDF derives PSK from `seed_family` using approved info string.
3. Component initiates (or accepts) Noise_IKpsk3 handshake with its peer.
4. `install_session_keys()` installs session keys into `CryptoTransport`.
5. Component registers with etcd as `READY`.
6. Normal pipeline operation begins.

**Failure policy:**
- If the peer is unreachable: retry up to 5 times with 2-second intervals.
- If all retries fail, or if PSK/key mismatch is detected: `std::terminate()`
  (fail-closed, consistent with ADR-013).

## Static Keypair Provisioning

`tools/provision.sh` is extended to generate a **dedicated X25519 static keypair**
per component for Noise_IKpsk3. These are separate from the Ed25519 keypairs used
for log signing (ADR-013). Key reuse across different cryptographic contexts is
explicitly prohibited.

```bash
# Extended provision.sh (ADR-024 addition)
# Per component: generate X25519 keypair for Noise_IKpsk3
sodium_keygen --type x25519 --out /etc/ml-defender/${component}/noise_static.key
```

Static public keys are stored in `deployment.yml` (ADR-021) as the Noise pre-message
payload, enabling each component to know its peer's static public key before the
handshake.

## Open Questions (to be resolved before implementation)

**OQ-5 — Static key revocation:**
If a physical node is stolen (hospital, school), how are its X25519 static keypairs
revoked without a PKI or CRL? Without a revocation mechanism, a stolen node can
participate in future handshakes indefinitely. Must define an emergency rotation
procedure before implementation. Candidate: Fingerprint Blocklist distributed via
etcd (Gemini); or emergency `tools/provision.sh` re-run across all nodes.

**OQ-6 — Key rotation on re-provisioning:**
If a component is re-provisioned and its static keypair changes, peers holding the
old public key in `deployment.yml` will reject the new handshake. What is the safe
rotation procedure? Must be resolved before fleet deployments.

**OQ-7 — Replay in first handshake message (updated):**
See "Note on replay" above. Decision: no timestamp required for v1 (PSK binding
sufficient given the threat model). Optional mitigation deferred to PHASE 3 evaluation.
Must document explicitly in threat model that first message replayability is accepted.

**OQ-8 — Performance on ARMv8 commodity hardware + Noise_KK comparison:**
Measure handshake latency and CPU on target hardware (~150–200 USD) before committing
to noise-c. Include explicit comparison of Noise_IKpsk3 vs Noise_KK:
- **Noise_KK** (Grok minority): simpler, better mutual authentication when all nodes
  have mutually pre-provisioned keypairs. Relevant for small closed deployments
  (hospital, school). Evaluate KCI (Key Compromise Impersonation) resistance of both.
- Decision: IKpsk3 for v1 (identity hiding benefit); KK reconsidered if OQ-8 shows
  meaningful advantage in closed deployments.

## Minority Positions (registered per Consejo methodology)

**ChatGPT — separate provisioning for Noise, no PSK from seed_family:**
Full domain separation at provisioning level. Not adopted. D4 (HKDF domain
separation via specific info string) is cryptographically sufficient with
libsodium used correctly.

**ChatGPT — adopt Noise_XX from the start:**
Maximum flexibility for future dynamic fleet scenarios. Not adopted. 1.5-RTT
overhead not justified for current closed deployments. Registered for fleet ADR.

**Grok — Noise_KK as alternative:**
Simpler, better mutual auth for closed deployments. Registered as part of OQ-8
evaluation. IKpsk3 confirmed for v1.

**Qwen — implement Noise_IKpsk3 directly over libsodium without noise-c:**
~2 days effort, saves ~150 KB binary footprint. Registered as part of R4/R10
evaluation. Both paths are valid; choice deferred to OQ-8 measurement.

**DeepSeek — metrics for noise-c acceptance (R10):**
Specific criteria (<200 KB, <50 ms) adopted and incorporated above.
